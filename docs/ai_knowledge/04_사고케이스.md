## 케이스 1: 수율 100% 초과 (122.5%) — 2026-05-15

### 증상
- 5/15 분석 → 공정별 현황 → 전처리 원육수율 **122.5%** 표시
- 산출(2,686 kg)이 투입(2,192 kg)보다 많은 비정상값

### 진단 과정
1. **물리적으로 불가능** — 산출 > 투입은 데이터/로직 문제
2. 분모(투입 KG) 또는 분자(산출 KG) 의심
3. DB 직접 확인: 5/15 방혈 8건 합 = 2,998.86 kg (125박스)
4. 화면 표시 = 2,192.92 kg (92박스) → **805.94 kg, 33박스 누락**
5. 전처리 record의 `wagons` 필드 확인: {1, 3, 5, 6, 7, 9}
6. 방혈 cart 8개 중 **cart 2, 8이 전처리 wagons에 안 적힘**
7. 사용자 확인: "실제로는 cart 2, 8도 전처리에 다 사용됨" → 입력 누락

### 근본 원인
**`analysis.js`의 wagons 매칭 룰이 너무 엄격.**
이전: `_ppWagons.includes(_normW(r.cart))` — wagons에 적힌 cart만 인정
실제: 작업자가 wagons에 일부 cart를 빠뜨려도 실제로는 사용됨

### 해결
**3개 함수 모두 룰 완화** (5/15):
- `analysis.js` 공정별 현황 (line ~2095)
- `getThKgByPP_` (월간 일보 원육 사용량)
- `getThByPartByPP_` (부위별)

새 룰: **"그날 end된 방혈 전체를 그날 전처리 투입으로 인정"**
```js
_rawTh = _candidates.filter(r => _endsOnDay(r,d));
```

폴백 유지: end 매칭 0건이면 wagons 매칭 → 옛 데이터 호환

### 효과
- 5/15 원육수율 69.7% → 51.0% (정상)
- 5/15 전처리 원육수율 122.5% → 89.6% (정상)

### 교훈
- **wagons 입력 누락에 관대하게** — 작업자 실수 보호
- **end 시간 매칭이 더 신뢰할 수 있는 지표** (실제 작업 완료 시점)

## 케이스 2: 작업인원이 디바이스마다 다르게 계산 — 2026-05-14

### 증상
- 출퇴근 화면: 28명 표시
- 월간 일보: 작업인원 29명 (1명 더 카운트)
- 다른 PC에서 추가한 직원("이유진")이 이 PC 화면엔 안 보임

### 진단 과정
1. attendance DB 확인 → 5/14 record 29명 (checkin 태그)
2. 직원 마스터 코드 확인 → `localStorage.getItem('att_employees_v1')`
3. **localStorage = 디바이스별 분리!** 디바이스 간 불일치 발생
4. 다른 PC: 이유진 추가 → localStorage에만 저장 → attendance엔 잔존 → 이 PC 화면엔 안 보임

### 근본 원인
**localStorage 사용 (다중 디바이스 동기화 원칙 위반)**
- 운영 룰: "localStorage 키/설정 저장 금지 (디바이스별 분리됨)"
- 직원 마스터 = 모든 디바이스 공유 필요 → Firestore에 있어야 함

### 해결
**`attendance.js` 직원 마스터 → Firestore 공유로 전환:**
- 저장 위치: `_config/attendance_employees`
- 로딩 순서: Firestore → localStorage(폴백) → DEFAULT
- 저장: Firestore + localStorage(캐시) 동시
- 모든 디바이스 자동 동기화

### 추가 해결: QC 분류
- 이유진 = QC → 작업인원에서 제외해야 함
- 직원 객체에 `role` 필드 추가:
  - `production`: 카운트 포함
  - `qc`, `manager`: 카운트 제외
- `_moFetchNonProductionEmps()` 함수가 마스터에서 role 읽고 카운트 시 제외

### 교훈
- **localStorage는 절대 사용 금지** — 모든 설정은 Firestore
- 직원/장비/제품 같은 마스터 데이터는 무조건 공유 저장소
- role 같은 분류 필드로 카운트 로직 유연하게

## 케이스 3: thawing.date가 시작일로 저장됨 — 2026-05-14

### 증상
- 5/14 thawing 8건의 `date` 필드가 5/14 (시작일)로 저장됨
- 룰: date = end 날짜 = 작업일 = 5/15 가 정답
- 이력추적, 입고관리 등에서 데이터 불일치

### 진단 과정
1. DB 직접 확인 → 8건 모두 date = 5/14 (잘못)
2. 새 thawing 저장 시 클라이언트가 보낸 date 그대로 저장된 듯
3. `common.js` `fbSave` 함수 확인
4. line 522-524: `date = 시작일` 로 덮어쓰는 코드 발견

### 근본 원인
**fbSave가 클라이언트 값을 신뢰**
- 옛 룰에서는 date = 시작일이었는데, 룰 바뀌면서 클라이언트는 종료일 보냄
- 하지만 fbSave에서 또 시작일로 덮어씀 (수정 누락)

### 해결
**fbSave에서 date 강제 정정:**
```js
// 룰: date = 시작일 + 1일 = 종료일
const startDate = String(d.start||'').slice(0,10);
d.date = addDays(startDate, 1);
// docId도 종료일 prefix로 재생성
const docId = `th_${d.date.replace(/-/g,'')}_*`;
```

DB 정정: 잘못 저장된 5/14 thawing 8건 → date=5/15로 PATCH 완료.

### 교훈
- **저장 시점에 룰 강제 적용** — 클라이언트 값 신뢰하지 말 것
- 룰 변경 시 저장/조회/표시 3곳 모두 점검
- 이미 잘못 저장된 데이터는 DB 직접 정정

## 케이스 4: 부동소수점 오차 표시 — 2026-05-15

### 증상
- 이력추적 자숙 KG 칸에 `444.7000000000000005` 표시
- 다른 KG 칸도 비슷한 패턴

### 진단
- JS의 float 합산 부동소수점 오차
- DB에는 정상 저장됐는데 화면 표시 시 raw 값 그대로 출력

### 해결
- `trace.js`의 KG 표시 셀 (자숙/방혈/파쇄) 모두 `r2(parseFloat(...))` 적용
- `r2()` = 소수점 1자리 반올림 헬퍼

### 교훈
- **숫자 표시 시 항상 반올림** — raw 값 그대로 노출 금지
- 합산되는 모든 숫자는 표시 시점에 정리

## 케이스 5: 진행중(packing_pending) 데이터가 차트에 섞임 — 2026-05-12

### 증상
- 월별현황 차트에 미완료 작업이 포함되어 수치가 부정확
- "오늘 작업 진행중인데 왜 차트에 다 잡혀?"

### 진단
- `packing_pending` = 임시 저장 (작업 미완료)
- 차트는 packing + packing_pending 둘 다 합산했음
- 진행중 데이터 = 부정확한 데이터

### 해결
**룰: packing_pending에 record 있는 날 = 차트에서 제외**
- 월별현황 (`monthly_production.js` + `analysis.js`): 제외
- AI 기간 분석: 제외
- 일별실적/이력추적/일별요약/챗봇: 그대로 (현재 상태 보여야 함)

`_mpPendingDates` 변수로 진행중 날짜 set 유지, 렌더 시 필터.

### 교훈
- **분석 차트 = 완료된 데이터만** — 진행중은 제외
- 표시 목적별로 필터 다르게 적용 (분석 vs 실시간)

### 룰 (반복 강조)

- 모든 설정/마스터/상태는 Firestore 공유 저장소

## 사고 진단 일반 흐름

새 사고 발생 시 따를 절차:

1. **증상 정확히 묘사** (사용자 표현 그대로)
2. **DB 직접 확인** (운영 데이터 실제 값)
3. **물리적으로 가능한지 검토** (수율 100% 초과 등 = 즉시 의심)
4. **분자/분모 추적** (수율 문제 시)
5. **코드의 매칭/필터 룰 확인** (가장 흔한 원인)
6. **다른 디바이스 동기화 확인** (localStorage 의심)
7. **저장/조회/표시 3곳 모두 점검** (룰 변경 시)
8. **근본 원인 + 즉시 해결 + 향후 예방** 동시 처리
